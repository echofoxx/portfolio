You are a principal product architect, enterprise portfolio-management expert, senior UX designer, security engineer, data architect, and full-stack software development team.
Build a comprehensive, functional, Docker-deployable enterprise Portfolio Management Tool based on the two attached DDC5I requirements artifacts:

1. DDC5I Enterprise Portfolio Requirements Draft 1.2
2. DDC5I Requirements Traceability Matrix Draft 1.2
The resulting application must be usable immediately after the first build. Do not produce only wireframes, static mockups, screenshots, or disconnected pages. ============================================================

1. PRODUCT MISSION ============================================================
Create a unified DDC5I Enterprise Demand, Portfolio, Project, Resource, Financial, Investment, Outcome, and Benefits Management platform.
The application must connect:
Mission and strategy → core functions and organizational responsibilities → demand and work intake → assessment and prioritization → leadership decisions → portfolio and program management → project execution → resources and skills → budgets and investments → risks, dependencies, and decisions → outcomes and benefits → executive reporting and the DDC5I story.
The organizational hierarchy must initially include:

* DDC5I Enterprise
* JAD
* DSD
* AID
* CID
* JFID
* C3OD2
The hierarchy must remain configurable for branches, teams, shared services, cross-division initiatives, and future organizational changes.
============================================================ 2. PRODUCT BOUNDARY AND ARCHITECTURE PRINCIPLES
Treat the application as a vendor-neutral operational reference implementation that can operate independently for the MVP and later integrate with:

* ServiceNow Strategic Portfolio Management
* ProjectOS
* Microsoft Outlook and Microsoft 365
* SharePoint
* Power BI
* Advana or War Data Platform
* Financial systems
* Workforce and personnel systems
* Enterprise identity providers
* Jira
* Azure DevOps
* Enterprise records-management services
Architect the application so that:

1. It is fully usable before any external integration is connected.
2. Every integrated field has one authoritative owner.
3. The system never allows ServiceNow, ProjectOS, or this application to independently overwrite the same authoritative field without a defined synchronization contract.
4. Stable canonical identifiers are used for organizations, missions, demands, projects, milestones, tasks, resources, financial records, risks, decisions, benefits, and integrations.
5. Integrations use documented APIs, events, retries, reconciliation, error handling, audit records, and source lineage.
6. Configuration is preferred over custom code.
7. The MVP is implemented as a modular monolith that can later be separated into services without redesigning the core data model.
8. No critical feature depends on an internet connection or an external CDN.
Do not claim that all 307 requirements are implemented unless they actually are. Import or represent the RTM in the application and classify each requirement as:

* Implemented
* Partially implemented
* Planned
* Deferred
* Requires integration
* Requires policy or governance decision

Use a modern, maintainable, locally deployable stack.

* Background-job abstraction for notifications, report generation, imports, reconciliations, and scheduled checks
* Docker and Docker Compose
* Automated database migration and seed process
* Local file-storage adapter using a Docker volume, with a future object-storage adapter
* Local demonstration authentication for development, with an interface for future OIDC, SAML, CAC, or PIV integration
A comparable stack may be used only when it preserves the same deployment, security, maintainability, accessibility, testing, and integration capabilities.

docker compose up -d --build
The application must automatically:

* Create the database
* Run migrations
* Seed demonstration data
* Create demonstration users
* Pass container health checks
* Expose the web interface on a clearly documented port
============================================================ 4. NON-NEGOTIABLE FIRST-BUILD EXPERIENCE
On the first successful build, a user must be able to:

1. Sign in using documented demonstration accounts.
2. Open a populated DDC5I senior-leadership dashboard.
3. switch between enterprise and division views.
4. Submit a new demand using a guided intake process.
5. Save the demand as a draft.
6. Validate and submit it.
7. Triage it and request clarification.
8. Assess and score it.
9. Compare it with other demands.
10. Route it for approval.
11. Record a decision with rationale and conditions.
12. Convert an approved demand into a project without re-entering approved data.
13. Create project milestones, tasks, risks, issues, dependencies, and actions.
14. Update project status.
15. See changes automatically roll up to division and enterprise dashboards.
16. Drill from an executive metric into the contributing source records.
17. Filter, sort, search, export, and save views.
18. View an audit trail showing who changed what and when.
19. Import validated records from an Excel template.
20. Receive row-level import success, warning, and error results.
No visible button should be decorative. Every visible action must either work or be omitted from the MVP interface.
Use actual PostgreSQL persistence. Do not depend on browser localStorage as the primary data store.
